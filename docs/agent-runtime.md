# 智能体运行时

## 当前运行流程

入口：

```text
POST /api/runs
```

流程：

1. 根据 `agent_id` 读取智能体配置。
2. 根据用户输入检索智能体绑定的知识库，并过滤低质量和低相关片段。
3. 根据用户输入和工具绑定情况调用应用工具。
4. 拼接系统提示词、多轮历史、用户问题、知识库上下文、工具结果。
5. 调用模型网关生成回答。
6. 保存运行记录和执行步骤。
7. 返回回答、引用来源、执行过程和模型信息。

## 上下文结构

后端会把上下文组织成：

```text
用户问题：
...

知识库上下文：
...

应用工具结果：
...

请给出清晰回答。优先使用应用工具结果回答实时业务数据；
使用知识库内容时说明来源；如果没有可靠依据，请明确说明无法确认。
```

## 执行步骤

当前返回的步骤包括：

- 接收任务
- 知识库检索
- 应用工具调用
- 大模型生成

## 错误处理

当真实大模型调用失败时，后端会返回错误说明，提示用户检查：

- 模型接口地址
- API Token
- 模型名
- 后端服务

流式运行失败时，run 记录的 status 设为 failed，error 字段记录异常信息。前端运行详情页展示失败原因和重新执行按钮。

## 检索增强

- PDF 上传会先提取可读文本，再切块入库。
- 切块入库前会做可读性检测，丢弃二进制残留和乱码片段。
- 检索后会按最低相关度过滤结果，并对重复片段去重。
- 对中文短语加入关键词匹配兜底，降低本地 fallback embedding 对中文短语不稳定的影响。
- 如果没有可靠命中，知识库上下文为空，模型应明确说明无法确认，而不是基于弱相关内容回答。

## 前台上下文

`/portal` 会把最近会话和多轮上下文保存到浏览器 `localStorage`。刷新页面后会恢复消息列表和上下文，继续发送给后端用于多轮回答。

## 对话记忆管理

前台 `/portal` 的对话记忆通过 `conversations` 表持久化：

- 每个会话由前端生成的 `session_id` 标识。
- 完整消息列表存储在 `messages` 字段（JSON）。
- 上下文压缩后的摘要存储在 `summary` 字段，`summary_to_turn` 记录摘要覆盖到第几轮。
- 支持按智能体过滤会话列表、搜索会话内容、删除会话。

## 反馈机制

用户可对智能体回答进行点赞/点踩：

- 反馈记录存储在 `feedbacks` 表，关联 `run_id` 和 `agent_id`。
- 支持匿名提交反馈（前台用户无需登录）。
- 后台可查看反馈列表和按智能体统计的点赞/点踩数量。

## 已实现的增强

- Run 列表和 Run 详情页已完成。
- 引用来源和执行步骤已可在运行历史中回看。
- 流式输出已接入。
- 工具调用参数由模型或规则生成。
- Run 详情页展示每一步耗时。
- 工具调用结果单独入库并在运行详情中展示。
- 失败重试：POST /runs/{run_id}/retry 接口，前端运行详情页展示"重新执行"按钮。
- 流式运行失败时正确记录 status=failed 和 error 信息。
- 工具调用日志：每次工具调用记录到 tool_calls 表，含耗时、状态码、输入输出。

- WebSocket 实时通信：支持双向通信、自动重连、心跳保活、中途取消生成、工具调用进度推送

## 多 Agent 协作（v3.0）

主智能体可以绑定若干子智能体，由运行时在生成回答后解析调用标签并递归执行子智能体，将子智能体的回答整合到主回答中。

### 配置方式

在智能体配置中通过 `sub_agent_ids` 绑定子智能体：

```json
{
  "name": "客服主智能体",
  "system_prompt": "你是客服主智能体，复杂问题可调用子智能体协助。",
  "sub_agent_ids": ["子智能体A的ID", "子智能体B的ID"]
}
```

前端智能体编辑页提供子智能体多选 UI，候选列表为已发布智能体，并自动排除当前编辑的智能体自身以避免循环引用。

### 运行时流程

1. **注入子智能体说明**：构建 `system_prompt` 时，若当前递归深度 `_depth < 3` 且绑定了子智能体，运行时会查询每个子智能体的名称和描述，拼接到系统提示词中，告知主智能体可调用的子智能体列表和调用方式。
2. **LLM 生成回答**：主智能体正常调用 LLM 生成回答，回答中可包含 `<call_agent:子agent_id>问题</call_agent>` 标签来触发子智能体调用。
3. **解析调用标签**：LLM 生成完成后，运行时使用正则 `<call_agent:([^>]+)>(.*?)</call_agent>` 解析回答中的所有调用标签。
4. **递归调用子智能体**：对每个匹配的标签，调用子智能体的 `run` 方法（`_depth + 1`），将子智能体的回答追加到主回答中。
5. **结果整合**：子智能体的回答以 `[子智能体「名称」的回答]` 的形式追加到原标签位置之后。

### 调用标签示例

主智能体生成的回答：

```text
用户问的是订单物流和退款政策，我分别调用相关子智能体协助。

<call_agent:order-agent>查询订单 SF123456 的物流状态</call_agent>

<call_agent:policy-agent>退款政策是什么</call_agent>
```

运行时解析后会替换为：

```text
用户问的是订单物流和退款政策，我分别调用相关子智能体协助。

<call_agent:order-agent>查询订单 SF123456 的物流状态</call_agent>
[子智能体「订单查询」的回答]
订单 SF123456 当前状态：运输中，预计明日送达。

<call_agent:policy-agent>退款政策是什么</call_agent>
[子智能体「售后政策」的回答]
7 天无理由退款，需保留商品完好...
```

### 递归深度限制

为防止子智能体之间相互调用导致无限循环，运行时限制最大递归深度为 **3 层**：

| 深度 | 行为 |
|------|------|
| `_depth = 0` | 主智能体，可调用子智能体 |
| `_depth = 1` | 一级子智能体，可继续调用子智能体 |
| `_depth = 2` | 二级子智能体，可继续调用子智能体 |
| `_depth >= 3` | 不再注入子智能体调用说明，也不再解析调用标签 |

### 流式输出支持

流式运行（`stream` 方法）同样支持子智能体调用：LLM 生成完成后，解析调用标签并执行子智能体，子智能体的回答以追加 token 的形式推送给前端。

### 执行步骤记录

每次子智能体调用会记录到运行步骤中：

| 步骤名 | 状态 | 详情示例 |
|--------|------|----------|
| 子智能体调用 | 完成 | 调用子智能体「订单查询」(ID: xxx)，问题：查询订单... |
| 子智能体调用 | 失败 | 调用子智能体「订单查询」失败：连接超时 |

## 工具调用链编排（v3.0）

除了按关键词触发的单工具调用外，v3.0 支持配置工具链（`tool_chain`），按预设顺序串行调用多个工具，前序工具的输出可注入后序工具的输入。

### 配置格式

`tool_chain` 是一个有序列表，每个元素定义一个工具调用步骤：

```json
[
  {
    "tool_id": "工具A的ID",
    "input_mapping": { "query": "$user_input" }
  },
  {
    "tool_id": "工具B的ID",
    "input_mapping": { "query": "$prev_output" }
  }
]
```

### input_mapping 映射规则

| 映射值 | 含义 |
|--------|------|
| `$user_input` | 用户原始输入 |
| `$prev_output` | 前一个工具的输出 |
| 其他字符串 | 作为字面量直接传入 |

### 运行时执行流程

1. **执行时机**：工具链在 LLM 生成**之前**执行，结果整合到 `tool_context` 中作为上下文提供给模型。
2. **串行调用**：按 `tool_chain` 列表顺序依次调用每个工具，维护 `prev_output` 变量记录上一个工具的输出。
3. **输入映射**：根据每步的 `input_mapping` 构造工具输入，将 `$user_input` 或 `$prev_output` 替换为实际值。
4. **失败不中断**：单个工具调用失败时，记录失败原因作为该步的输出，但**不中断**后续步骤的执行。
5. **结果整合**：所有工具链结果以 `工具链：{工具名}\n结果：\n{内容}` 的格式拼接到 `tool_context`。

### 执行步骤记录

工具链的每一步都会记录到运行步骤：

| 步骤名 | 状态 | 详情示例 |
|--------|------|----------|
| 工具链调用-1 | 完成 | 调用工具「订单查询」(ID: xxx) |
| 工具链调用-2 | 失败 | 调用工具「物流查询」(ID: xxx) |

### 配置约束

- 工具链最多支持 **10 个步骤**（由 schema 校验）。
- 工具链中的工具必须已在 `tool_ids`（已绑定工具）中，前端会同步清理已解绑的工具步骤。

## 文件上传对话（v3.0）

前台支持在对话中上传文件，后端解析提取文本后注入对话上下文，让智能体能够即时分析文件内容。

### 支持的文件类型

| 类型 | 扩展名 | 处理方式 |
|------|--------|----------|
| PDF | `.pdf` | 提取可读文本 |
| 文本 | `.txt` | 直接读取文本 |
| Markdown | `.md` | 直接读取文本 |
| 图片 | `.png` `.jpg` `.jpeg` | 不提取文本，仅记录文件信息 |

### 上传接口

```text
POST /api/files/upload
Content-Type: multipart/form-data
```

- 文件大小限制：10MB
- 文本内容截断：8000 字符（避免上下文过长）
- 无需鉴权（前台匿名可访问）

返回结果：

```json
{
  "file_id": "文件唯一ID",
  "filename": "报告.pdf",
  "content_type": "pdf",
  "text_content": "提取出的文本内容...",
  "char_count": 3200
}
```

### 运行时注入

`AgentRuntime` 的 `run` 和 `stream` 方法均增加 `file_content` 参数。文件内容作为用户消息的一部分传给 LLM：

```text
用户问题：{user_input}

知识库上下文：
...

应用工具结果：
...

用户上传文件内容：
{file_content}

请给出清晰、简洁、可追溯的回答。
```

### 前端交互

- 用户在输入框旁选择文件后，前端调用 `/files/upload` 上传并获取文本内容。
- 发送消息时将 `file_content` 一并传入 `/runs/stream` 或 WebSocket。
- 发送完成后清空已上传文件。

## 上下文压缩（v3.0）

随着对话轮次增加，前端传入的 `messages` 列表会持续增长。运行时采用 **Token 估算 + 滑动窗口 + 摘要压缩** 策略控制上下文长度。

### 压缩策略

```text
压缩前（20 轮对话）:
[sys] [u1][a1] [u2][a2] ... [u14][a14] [u15][a15] ... [u20][a20]

压缩后:
[sys] [摘要: 第1-14轮的关键信息] [u15][a15] ... [u20][a20]
       ↑ LLM 生成的摘要              ↑ 最近6轮保留原文
```

### 配置项

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `context_token_threshold` | 6000 | 历史消息 Token 估算超过此值时触发压缩 |
| `recent_turns` | 6 | 压缩时保留最近几轮对话原文（每轮 2 条消息） |

### Token 估算

运行时使用粗估算法估算消息列表的 Token 数：

- 中文字符：约 1 字 ≈ 1.5 Token
- 其他字符：约 4 字符 ≈ 1 Token

### 压缩流程

1. 在 `_build_messages` 中估算历史消息的总 Token 数。
2. 若超过阈值 `6000`，分离早期消息和近期消息（最近 6 轮 = 12 条消息）。
3. 调用 LLM 对早期消息生成摘要。
4. 用摘要替换早期消息，作为 system 角色注入：`以下是之前对话的摘要，供你参考：{summary}`。
5. 摘要生成失败时，回退到只保留最近消息。

### 前端感知

运行结果中返回 `compressed` 字段，前端据此显示"上下文已压缩"提示，让用户感知压缩行为。前端本地仍保留完整消息列表，压缩仅影响发送给 LLM 的内容。

## WebSocket 流式通信（v3.0）

v3.0 新增 WebSocket 端点作为 SSE 的替代方案，支持双向通信和长连接。

### 端点

```text
WebSocket /ws/chat
```

### 客户端发送的消息

| 消息类型 | 格式 | 说明 |
|----------|------|------|
| 聊天请求 | `{"agent_id": "xxx", "input": "问题", "messages": [...], "file_content": ""}` | 发起一次流式对话 |
| 取消请求 | `{"action": "cancel"}` | 中途取消当前生成 |
| 心跳 | `{"type": "ping"}` | 心跳保活 |

### 服务端推送的事件

| 事件 | 数据 | 说明 |
|------|------|------|
| `tool_progress` | `{"tool_names": [...], "status": "running\|done"}` | 工具调用进度 |
| `steps` | `[...]` | 执行步骤 |
| `citations` | `[...]` | 引用来源 |
| `token` | `{"token": "..."}` | 流式 token |
| `cancelled` | `{"message": "..."}` | 生成已取消 |
| `error` | `{"message": "..."}` | 错误信息 |
| `done` | `{"run_id": "...", "model": "...", "duration_ms": 123}` | 生成完成 |
| `pong` | `{}` | 心跳响应 |

### SSE 事件转换

WebSocket 端点复用 `stream` 方法的 SSE 输出，通过 `parse_sse_to_ws` 函数将 SSE 格式转换为 WebSocket JSON 消息：

- 带事件名的 SSE（`event: steps\ndata: [...]`）→ `{"event": "steps", "data": [...]}`
- 仅数据的 SSE（`data: {"token": "..."}`）→ `{"event": "token", "data": {"token": "..."}}`

### 前端能力

- **模式切换**：前端可在 SSE 模式和 WebSocket 模式间切换，默认使用 SSE 保持兼容。
- **自动重连**：断开后最多重连 5 次，间隔 3 秒。
- **心跳保活**：每 30 秒发送一次 `ping`。
- **中途取消**：通过发送 `{"action": "cancel"}` 取消当前生成。
- **工具调用进度**：实时显示"正在调用工具..."提示。
