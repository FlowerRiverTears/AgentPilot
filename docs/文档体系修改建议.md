# AgentPilot 文档体系企业级专业修改建议

> 基于对项目全部 15 份文档的审阅，从企业级产品视角提出系统性改进建议。

---

## 一、总体评价

当前文档体系覆盖了项目从规划到落地的完整链路，内容详实、结构清晰，在开源项目中属于中上水平。但以企业级标准衡量，存在以下系统性问题：

1. **信息冗余严重** — 同一内容在 5+ 份文档中重复出现，且版本不一致
2. **文档与代码不同步** — 多处文档描述的功能状态与实际代码不符
3. **缺少关键运维文档** — 无监控、告警、备份恢复、故障排查手册
4. **安全文档缺失** — 无安全架构、威胁模型、数据分级、合规说明
5. **API 文档不够规范** — 缺少标准 OpenAPI 规范，响应码、错误码定义不完整
6. **架构决策未记录** — 缺少 ADR（Architecture Decision Record）

## 四、企业级安全改进建议

### 4.1 认证安全

| 当前状态 | 风险 | 建议 |
|----------|------|------|
| JWT Secret 硬编码为 `agentpilot-dev-secret-change-in-production` | 生产环境 Secret 泄露 | 强制从环境变量读取，启动时检查是否为默认值 |
| 默认管理员密码 `admin123` | 暴力破解风险 | 首次登录强制修改密码 |
| Token 有效期 24 小时 | 过长，被盗后影响大 | 缩短至 2-4 小时，引入 Refresh Token |
| 无登录失败锁定 | 暴力破解风险 | 5 次失败后锁定 15 分钟 |
| 无密码复杂度要求 | 弱密码风险 | 最少 8 位，含大小写和数字 |
| `auth_enabled=False` 可关闭鉴权 | 生产环境误用 | 仅在开发环境生效，生产环境强制开启 |

### 4.2 数据安全

| 当前状态 | 风险 | 建议 |
|----------|------|------|
| `model_configs.api_key` 明文存储 | 数据库泄露导致 API Key 泄露 | 使用 AES-256 加密存储 |
| 工具配置中 Headers 可能含敏感 Token | 日志或界面泄露 | 脱敏显示，加密存储 |
| 无审计日志 | 无法追溯操作 | 新增 audit_logs 表，记录所有写操作 |
| 无数据备份策略 | 数据丢失 | 补充 pg_dump 定时备份方案 |
| 前台匿名用户可无限调用 | 资源滥用 | 添加 Rate Limiting |

### 4.3 网络安全

| 当前状态 | 风险 | 建议 |
|----------|------|------|
| 无 HTTPS 强制 | 中间人攻击 | 生产环境强制 HTTPS |
| 无 CORS 策略说明 | 跨域攻击 | 文档说明 CORS 配置 |
| 无 CSP 策略 | XSS | 补充 Content-Security-Policy 配置 |
| Docker 端口直接暴露 | 攻击面过大 | 生产环境通过 Nginx 反向代理，不直接暴露 15432/16379 |

## 五、企业级架构改进建议

### 5.1 数据库迁移

| 当前状态 | 风险 | 建议 |
|----------|------|------|
| 手动 `init.sql` + 应用启动时 `_ensure_*` | 迁移不可控、不可回滚 | 引入 Alembic，每次 Schema 变更生成迁移脚本 |
| 无迁移版本记录 | 无法确认当前数据库版本 | Alembic 自动维护 `alembic_version` 表 |
| 无回滚能力 | 迁移失败无法恢复 | Alembic 支持 downgrade |

### 5.2 可观测性

| 当前状态 | 风险 | 建议 |
|----------|------|------|
| 无结构化日志 | 排查困难 | 统一 JSON 日志格式，包含 trace_id |
| 无指标采集 | 无法感知系统健康 | 引入 Prometheus metrics（请求延迟、错误率、队列深度） |
| 无分布式追踪 | 跨服务调用链不可见 | 引入 OpenTelemetry |
| 无健康检查深度 | `/health` 只返回 ok | 补充 DB、Redis、MinIO 连通性检查 |

### 5.3 部署与运维

| 当前状态 | 风险 | 建议 |
|----------|------|------|
| 单副本部署 | 无高可用 | 补充多副本部署方案 |
| 无资源限制 | OOM 风险 | Docker Compose 添加 memory/cpu limits |
| 无日志聚合 | 多容器日志分散 | 引入 ELK/Loki 日志聚合 |
| 无配置中心 | 配置分散在环境变量 | 考虑集中配置管理 |
| 无优雅关闭 | 请求中断 | FastAPI 添加 shutdown hook，等待进行中请求完成 |

## 七、文档质量标准建议

为避免文档再次腐化，建议建立以下标准：

1. **单一信源原则**：每类信息只在一个文档中维护，其他文档通过链接引用
2. **文档审查机制**：PR 模板加入"是否需要更新文档"检查项
3. **文档版本化**：每份文档头部标注最后更新日期和对应代码版本
4. **自动化校验**：API 文档从代码生成，数据库文档从 Schema 生成
5. **定期审计**：每季度审查一次文档与代码的一致性
