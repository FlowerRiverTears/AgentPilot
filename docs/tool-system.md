# 工具系统设计

## 定位

工具用于连接实时业务系统，不应该替代知识库。

适合工具的场景：

- 订单查询
- 物流查询
- 库存查询
- 用户信息查询
- 工单创建
- 数据库查询
- 当前时间

适合知识库的场景：

- FAQ
- 产品手册
- 售后政策
- 内部制度
- 培训资料

## 当前实现

工具注册位置：

```text
backend/app/tools/registry.py
```

内置工具：

- `current_time`
- `order_lookup`

数据库持久化工具：

```text
backend/app/repositories/tools.py
```

接口：

```text
GET    /api/tools           工具列表
POST   /api/tools           创建工具
GET    /api/tools/{id}      工具详情
PUT    /api/tools/{id}      更新工具
DELETE /api/tools/{id}      删除工具
POST   /api/tools/{id}/test 工具测试
```

当前已支持：

- 工具表 `tools` 持久化存储
- HTTP 工具配置（URL、Method、Headers、Query、Body、触发关键词、超时）
- 工具 CRUD（创建、读取、更新、删除）
- 工具测试接口（发送实际 HTTP 请求，返回状态码、耗时和输出）
- Agent 绑定数据库工具
- 运行时按关键词触发工具调用
- 内置工具作为 fallback，数据库工具优先

## 工具调用日志

每次工具调用自动记录到 `tool_calls` 表：

| 字段 | 说明 |
|------|------|
| `run_id` | 关联的运行 ID |
| `tool_id` | 关联的工具 ID |
| `tool_name` | 工具名称 |
| `input` | 调用输入 |
| `output` | 调用输出 |
| `status` | 调用状态（success / failed） |
| `status_code` | HTTP 状态码 |
| `elapsed_ms` | 耗时（毫秒） |
| `error` | 错误信息 |
| `detail` | 详细信息（JSON） |

查询接口：`GET /tools/calls?limit=50`（需鉴权）

## 当前限制

- 工具管理页面已具备基础创建、编辑、删除和测试能力，后续需要优化交互体验。
- 还没有 SQL 工具和鉴权配置页。
- 工具配置中的 Headers 可能包含敏感信息（如 Bearer Token），建议在后续版本中支持加密存储（当前仅对 `model_configs.api_key` 做了加密）。

### 工具测试

工具管理页应支持：

- 输入测试参数。
- 调用工具。
- 查看请求和响应。
- 查看耗时。
- 查看错误原因。

## 调用流程

1. Agent 绑定工具。
2. 用户提问。
3. 运行时判断是否需要工具。
4. 生成工具参数。
5. 调用工具。
6. 保存工具调用记录。
7. 将工具结果交给大模型生成回答。

## 安全原则

- SQL 工具默认只允许只读查询。
- HTTP 工具不应明文展示敏感 Header。
- 工具测试和调用应记录日志。
- 后台工具管理需要管理员权限。

## 工具调用链编排（v3.0）

除了按关键词触发的单工具调用外，v3.0 支持为智能体配置工具链（`tool_chain`），按预设顺序串行调用多个工具，前序工具的输出可注入后序工具的输入。

### 配置格式

`tool_chain` 是智能体配置中的一个有序列表，每个元素定义一个工具调用步骤：

```json
[
  {
    "tool_id": "工具A的ID",
    "input_mapping": { "query": "$user_input" }
  },
  {
    "tool_id": "工具B的ID",
    "input_mapping": { "query": "$prev_output" }
  }
]
```

### input_mapping 映射规则

`input_mapping` 是一个键值对字典，键为工具入参字段名，值为映射表达式：

| 映射值 | 含义 | 示例场景 |
|--------|------|----------|
| `$user_input` | 用户原始输入 | 第一步工具通常用此映射 |
| `$prev_output` | 前一个工具的输出 | 串联工具，将上一步结果作为下一步输入 |
| 其他字符串 | 作为字面量直接传入 | 固定参数，如 `{"type": "order"}` |

### 串行执行流程

1. 运行时按 `tool_chain` 列表顺序依次调用每个工具。
2. 维护 `prev_output` 变量，记录上一个工具的输出内容。
3. 对每一步，根据 `input_mapping` 构造工具输入：
   - 遇到 `$user_input` 替换为用户原始输入。
   - 遇到 `$prev_output` 替换为前一个工具的输出。
   - 其他值作为字面量直接传入。
4. 调用工具的测试接口（`test_tool`）执行实际 HTTP 请求。
5. 将当前工具的输出更新到 `prev_output`，供下一步使用。

执行示意：

```text
用户输入 "查询订单 SF123 的物流"
  ↓
步骤1：订单查询工具
  input_mapping: { "query": "$user_input" }
  实际输入: { "query": "查询订单 SF123 的物流" }
  输出: "订单 SF123 已发货，运单号 SF123456"
  ↓ prev_output = "订单 SF123 已发货，运单号 SF123456"
步骤2：物流查询工具
  input_mapping: { "query": "$prev_output" }
  实际输入: { "query": "订单 SF123 已发货，运单号 SF123456" }
  输出: "运单 SF123456 当前在途，预计明日送达"
```

### 错误处理

工具链采用**失败不中断**策略：

| 情况 | 处理方式 |
|------|----------|
| 工具调用成功 | 输出作为 `prev_output` 传给下一步，状态记为"完成" |
| 工具调用失败 | 错误信息作为该步输出，状态记为"失败"，**继续执行下一步** |
| 工具不存在 | 输出"工具调用失败"，状态记为"失败"，继续执行下一步 |

这样设计保证链式调用中单个工具的故障不会阻塞后续工具的执行，最终所有工具结果（含失败原因）都会整合到上下文中供 LLM 参考。

### 结果整合

工具链的所有结果以如下格式拼接到 `tool_context`，作为应用工具结果的一部分提供给 LLM：

```text
工具链：订单查询
结果：
订单 SF123 已发货，运单号 SF123456

工具链：物流查询
结果：
运单 SF123456 当前在途，预计明日送达
```

### 前端工具链编排 UI

智能体编辑页提供可视化的工具链编排界面：

- **添加步骤**：从已绑定工具（`tool_ids`）的下拉列表中选择工具添加到链尾，默认输入映射为 `{ "query": "$user_input" }`。
- **上下排序**：每个步骤提供"上移""下移"按钮，调整工具调用顺序。
- **输入映射选择**：每个步骤可配置 `input_mapping`，选择 `$user_input` 或 `$prev_output`。
- **移除步骤**：每个步骤可单独移除。
- **自动清理**：当已绑定工具变化时，自动清理工具链中已解绑的工具步骤。

### 配置约束

- 工具链最多支持 **10 个步骤**（由 `AgentCreate` schema 校验）。
- 工具链中的工具必须已在智能体的 `tool_ids`（已绑定工具）中。
- 工具链在 LLM 生成**之前**执行，结果作为上下文提供给模型。

## 工具调用统计（v3.0）

基于 `tool_calls` 表记录的工具调用日志，系统提供工具调用统计能力，用于评估工具的运行状况和健康度。

### 数据来源

每次工具调用（含工具链中的每一步）都会记录到 `tool_calls` 表，包含以下可用于统计的字段：

| 字段 | 统计用途 |
|------|----------|
| `tool_id` | 按工具维度聚合统计 |
| `status` | 计算成功率（success / failed） |
| `elapsed_ms` | 计算平均耗时 |
| `created_at` | 按时间范围统计调用趋势 |

### 统计指标

| 指标 | 计算方式 |
|------|----------|
| 调用次数 | 按 `tool_id` 分组计数 |
| 成功率 | `status = "success"` 的记录数 / 总记录数 |
| 平均耗时 | 按 `tool_id` 分组对 `elapsed_ms` 求平均值 |

### 查询接口

工具调用日志查询接口：

```text
GET /api/tools/calls?limit=50&offset=0
```

- 需鉴权（登录用户可访问）。
- 返回最近的工具调用记录列表，按 `created_at` 倒序排列。
- 支持分页（`limit` 和 `offset` 参数）。

返回字段示例：

```json
{
  "id": "记录ID",
  "run_id": "关联的运行ID",
  "tool_id": "工具ID",
  "tool_name": "订单查询",
  "input": "查询订单 SF123",
  "output": "订单已发货...",
  "status": "success",
  "status_code": 200,
  "elapsed_ms": 320,
  "error": "",
  "created_at": "2026-06-21T10:30:00"
}
```

### 统计应用场景

- **工具健康度监控**：通过成功率识别频繁失败的工具，及时排查接口故障。
- **性能优化**：通过平均耗时定位慢工具，优化接口响应速度。
- **使用热度分析**：通过调用次数了解各工具的使用频率，指导工具维护优先级。
- **运行详情回看**：运行详情页展示每次运行关联的工具调用记录，含耗时、状态码和输入输出。
