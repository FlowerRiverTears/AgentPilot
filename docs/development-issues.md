# AgentPilot 开发问题记录

> 记录项目开发过程中遇到的关键问题、原因分析和解决方案。

---

## 一、PDF 解析

### 1.1 PDF 二进制内容泄漏到文本中

**问题**：PDF 文件解析后，提取的文本中混入了 PDF 内部的二进制结构标记（如 `%PDF-`、`endobj`、`obj <`、`endstream`、`xref`、`/FlateDecode`、`/XRef`），导致后续切块和检索中出现大量乱码。

**原因**：pypdf 在提取文本时，对于内嵌字体、编码流等部分无法正确解码，会将原始 PDF 结构语法当作文本输出。

**解决**：在 `text_quality.py` 中实现 `looks_like_pdf_binary_text()` 函数，检测 7 种 PDF 结构标记，当标记出现 ≥ 2 个或文本以 `%PDF-` 开头且包含 `obj` 时，判定为二进制残留并过滤。

---

### 1.2 PDF 单页提取失败导致整个文档不可用

**问题**：某些 PDF 的个别页面损坏或格式异常，直接提取会导致整个文档解析抛异常。

**原因**：PDF 文件可能部分损坏、加密或使用了不支持的压缩格式。

**解决**：对每一页的 `extract_text()` 单独 try-except，失败时将该页文本设为空字符串而非中断整个流程。图片提取同样做了容错处理。

---

### 1.3 扫描件/图片型 PDF 无文本可提取

**问题**：纯扫描件或图片型 PDF 没有可提取的文本，上传后不报错但知识库中没有任何有效内容，用户无法理解原因。

**原因**：扫描件 PDF 的文字以图像形式存在，pypdf 无法从图像中提取文字。

**解决**：当提取结果为空时抛出 `DocumentParseError`，明确提示"PDF 文件没有可提取的文本或图片，可能是扫描件或图片型 PDF"。

---

### 1.4 PDF 文件类型判断不够鲁棒

**问题**：仅靠文件扩展名判断文件类型不可靠，用户可能上传扩展名不匹配的文件。

**原因**：文件名可能被手动修改，或上传时 content-type 不准确。

**解决**：采用三重判断：`suffix == ".pdf" or content_type == "application/pdf" or raw.startswith(b"%PDF")`，即使扩展名和 content-type 都不对，也能通过文件头的 `%PDF` 魔数识别。

---

### 1.5 PDF 中提取的图片过小无意义

**问题**：PDF 中内嵌了大量极小的装饰性图片（如图标、分隔线等），这些图片没有检索价值，反而浪费存储和计算资源。

**原因**：PDF 规范中任何嵌入资源都是"图片"，不区分内容图片和装饰图片。

**解决**：过滤掉小于 1KB 的图片，只保留有实际内容价值的图片。

---

## 二、文本编码

### 2.1 非 UTF-8 编码的文本文件无法解析

**问题**：中文环境下大量文件使用 GBK/GB18030 编码，直接用 UTF-8 解码会报 `UnicodeDecodeError`，导致文件上传失败。

**原因**：Windows 环境下中文文本默认编码通常是 GBK/GB18030，而非 UTF-8。

**解决**：依次尝试 `utf-8-sig`（带 BOM 的 UTF-8）、`utf-8`、`gb18030` 三种编码解码，全部失败后用 `utf-8` + `errors="ignore"` 兜底，确保至少能返回部分内容。

---

### 2.2 文本中混入控制字符和空字符

**问题**：从 PDF 或其他来源提取的文本中包含 `\x00`（空字符）和其他不可见控制字符，影响后续文本处理和显示。

**原因**：二进制文件解析时，控制字符和空字符会混入文本流。

**解决**：实现 `_clean_text()` 函数，只保留 `\n\r\t` 和 ASCII ≥ 32 的字符，过滤掉 `\x00` 和其他控制字符。

---

## 三、文本质量过滤

### 3.1 Unicode 替换字符（U+FFFD）过多导致文本不可读

**问题**：编码转换失败产生的 `\ufffd`（Unicode 替换字符）大量出现在文本中，导致文本内容不可读。

**原因**：解码失败时使用替换策略，每个无法解码的字节都变成 `\ufffd`。

**解决**：当替换字符占比超过 1% 时判定为不可读文本，予以过滤。

---

### 3.2 控制字符过多导致文本不可读

**问题**：某些提取的文本中控制字符（如退格、响铃等）占比过高，实际可读内容很少。

**原因**：二进制数据被误当作文本提取。

**解决**：使用 `unicodedata.category()` 检测控制字符（类别以 `C` 开头），占比超过 2% 时判定为不可读。

---

### 3.3 符号/标点占比过高导致文本无实质内容

**问题**：某些切块几乎全是标点符号或特殊符号，没有实际文字内容。

**原因**：PDF 表格、页眉页脚等区域提取后只剩分隔符和符号。

**解决**：当符号/标点占比超过 45% 时判定为不可读文本。

---

### 3.4 字母/数字占比过低导致文本无实质内容

**问题**：某些切块虽然"可读字符"比例达标，但实际没有有意义的词语。

**原因**：装饰性文本、页码标记等。

**解决**：当字母/数字占比低于 25% 时判定为不可读。

---

### 3.5 非 ASCII 非 CJK 字符异常多（疑似乱码）

**问题**：某些文本中出现了大量非 ASCII、非 CJK 的 Unicode 字符（如西里尔字母、希伯来字母等），在中英文文档中这是乱码的典型特征。

**原因**：编码错误导致字节被错误映射到其他 Unicode 区段。

**解决**：两重检测：
- 可疑扩展字符占比超过 2% 且 CJK 占比低于 25% 时判定为不可读
- 非 ASCII 非 CJK 字符占比超过 5% 且 CJK 占比低于 5% 时判定为不可读

---

### 3.6 文本过短无检索价值

**问题**：极短的文本片段（如单独的页码"3"、孤立的标题）没有检索价值，还会稀释检索质量。

**原因**：切块边界恰好在段落中间，或文档本身有大量短行。

**解决**：当归一化后文本长度低于 20 时判定为不可读。

---

### 3.7 历史遗留的不可读切块残留在数据库中

**问题**：在文本质量过滤功能上线之前，已经入库了大量不可读的切块，这些切块仍然会出现在检索结果中。

**原因**：功能是后加的，已有数据未经过滤。

**解决**：在服务启动时执行 `_cleanup_unreadable_chunks()`，遍历所有文本切块，删除 `is_readable_text()` 返回 False 的记录。

---

## 四、Embedding 服务

### 4.1 远程 Embedding 服务不可用导致系统崩溃

**问题**：当远程 Embedding API 不可用时，文档上传和检索功能完全不可用。

**原因**：网络故障、API 服务宕机、配额耗尽等。

**解决**：远程调用异常时 fallback 到基于 SHA256 哈希的本地 stub 向量，确保系统不会因 Embedding 服务故障而完全不可用。多模态文本编码和图片编码也有相同的 fallback 逻辑。

---

### 4.2 Embedding 配置缺失时的降级处理

**问题**：用户可能没有配置独立的 Embedding 服务，或只配置了 LLM 服务。

**原因**：Embedding 服务是可选配置。

**解决**：`embedding_base_url` 和 `embedding_api_key` 分别 fallback 到 `llm_base_url` 和 `llm_api_key`；`embed_text()` 方法按优先级尝试：多模态 → 远程文本 → 本地 stub。

---

### 4.3 多模态 Embedding 未配置时图片向量化降级

**问题**：当多模态 Embedding 服务未配置时，图片无法向量化。

**原因**：多模态 Embedding 是可选功能。

**解决**：`embed_image()` 在多模态不可用时 fallback 到基于图片字节哈希的 stub 向量，并输出 warning 日志。

---

### 4.4 向量维度动态发现

**问题**：用户可能不清楚所用 Embedding 模型的输出维度，配置错误会导致向量维度不匹配。

**原因**：不同模型的维度不同（如 bge-m3 是 1024 维，CLIP 是 768 维）。

**解决**：在每次远程调用成功后，用实际返回的向量长度更新维度配置，自动适配模型输出。

---

### 4.5 文本和图片向量空间不一致

**问题**：文本用 bge-m3（1024 维）编码，图片用 CLIP（768 维）编码，维度不同导致 pgvector 无法在同一列存储和计算相似度，跨模态检索完全失效。

**原因**：两个不同模型的向量空间不兼容，无法直接比较余弦距离。

**解决**：多模态可用时，文本也使用多模态模型的文本编码器编码，确保文本和图片输出在同一向量空间。数据库 embedding 列维度根据是否配置多模态模型动态决定。

---

## 五、检索质量

### 5.1 纯向量检索对中文关键词匹配效果差

**问题**：语义向量检索在处理精确关键词查询时（如"SET NX EX"这种 Redis 命令），可能因为语义距离较远而漏掉高度相关的文档。

**原因**：向量编码侧重语义相似性，对精确词法匹配不敏感。

**解决**：实现 `lexical_relevance()` 函数作为词法匹配兜底，采用 bigram+trigram 的中文分词策略，对英文用空格分词。最终得分取 `max(vector_score, lexical_score)`，确保词法精确匹配不会被遗漏。

---

### 5.2 检索结果中存在重复内容

**问题**：由于切块 overlap 和多知识库检索，同一内容可能以不同切块形式出现多次。

**原因**：切块重叠导致边界内容重复；多知识库检索时各库可能包含相同文档。

**解决**：使用 `seen_contents` 集合对归一化后的内容去重。

---

### 5.3 低相关性检索结果污染上下文

**问题**：向量检索总是返回 top-k 结果，即使相关性很低，这些低质量上下文会误导 LLM 生成错误回答。

**原因**：向量检索基于距离排序，不保证结果与查询相关。

**解决**：定义 `retrieval_min_score=0.3` 和 `retrieval_min_lexical_score=0.3` 两个可配置阈值，当向量分数和词法分数都低于阈值时，跳过该切块。

---

## 六、切块策略

### 6.1 切块边界处丢失上下文

**问题**：固定大小切块在边界处会切断语义完整的句子或段落，导致边界处的信息在检索时无法被完整匹配。

**原因**：简单的固定窗口切块没有考虑语义边界。

**解决**：使用 `overlap=120` 的重叠切块策略，确保边界内容在相邻切块中都有出现。

---

### 6.2 不可读切块进入知识库

**问题**：即使经过文本清洗，某些切块仍然可能是乱码或无意义内容。

**原因**：清洗无法覆盖所有情况。

**解决**：在 `build_chunks()` 中对每个切块调用 `is_readable_text()` 过滤，只有通过质量检测的切块才会入库。

---

## 七、数据库迁移

### 7.1 数据库表结构演进需要向后兼容

**问题**：随着功能迭代，`document_chunks` 表需要新增多个字段，但已有部署的数据库没有这些字段，直接查询会报错。

**原因**：项目没有使用正式的数据库迁移工具（如 Alembic），需要手动处理 schema 演进。

**解决**：双重保障：
- `init.sql` 使用 `IF NOT EXISTS` 逐列检查并添加，确保幂等性
- 应用启动时执行 `_ensure_rag_standard_columns()`，作为应用层的迁移保障

---

### 7.2 Agent 的 model_config_id 字段需要回填

**问题**：早期创建的 Agent 没有 `model_config_id` 配置，升级后这些 Agent 无法正确路由到 LLM 模型。

**原因**：`model_config_id` 是后加的功能。

**解决**：在启动时执行 `_backfill_agent_model_configs()`，遍历所有 Agent，将缺失的 `model_config_id` 回填为默认配置 ID。

---

## 八、运行时容错

### 8.1 LLM 调用失败时用户看到无意义的错误

**问题**：当 LLM API 调用失败时，原始异常信息对用户毫无帮助。

**原因**：网络问题、API Key 错误、模型不存在等。

**解决**：捕获 `RuntimeError` 后返回结构化的错误提示，引导用户检查"接口地址、Token 和模型名"。

---

### 8.2 流式输出中断时的处理

**问题**：流式生成过程中 LLM 可能中途失败，前端已经收到部分 token，需要优雅地结束。

**原因**：网络不稳定、API 超时等。

**解决**：捕获异常后将 `generation_status` 设为"失败"，输出"流式生成失败，请检查模型配置"的提示，并继续完成 run 记录的创建。

---

### 8.3 工具调用结果的 fallback 机制

**问题**：数据库中配置的 HTTP 工具可能调用失败，需要降级到内置注册的工具。

**原因**：外部 API 不可用或配置错误。

**解决**：先尝试数据库配置的工具，如果无结果再 fallback 到内置注册的工具。

---

## 九、存储容错

### 9.1 MinIO 服务不可用时不应阻塞文档入库

**问题**：MinIO 服务可能未部署或不可用，如果图片上传失败导致整个文档入库流程中断，影响太大。

**原因**：MinIO 是可选组件。

**解决**：`_upload_images_to_minio()` 整体 try-except，失败时只输出 warning 日志，返回空的 `image_map`，文档入库流程继续进行（只是图片 URL 为空）。
