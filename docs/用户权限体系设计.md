# 用户权限体系设计文档

## 一、现状分析

### 1.1 当前权限数据分布

| 位置 | 文件 | 现状 |
|------|------|------|
| 用户模型 | `backend/app/models/user.py` | `role` 字段默认 `"admin"`，无角色枚举 |
| 认证 Schema | `backend/app/schemas/auth.py` | `LoginResponse.role` 默认 `"admin"`，`UserInfo` 包含 role |
| 安全模块 | `backend/app/core/security.py` | JWT 编码了 `role`，但后端从未根据 role 做权限判断 |
| 依赖注入 | `backend/app/core/deps.py` | `get_current_user` 只检查 `is_active`，不检查 `role` |
| 认证路由 | `backend/app/api/routes/auth.py` | 仅 login/logout/me，无用户管理接口 |
| 前端 Store | `frontend/src/stores/auth.ts` | 存储了 `role`，但未用于权限控制 |
| 前端路由 | `frontend/src/router/index.ts` | 仅判断是否登录，不区分角色 |

### 1.2 核心问题

1. **角色字段形同虚设** — `role` 只有一种值 `"admin"`，所有登录用户权限相同
2. **无用户管理功能** — 无法创建/编辑/禁用用户，无法分配角色
3. **无权限校验逻辑** — 后端路由不区分角色，任何登录用户都能执行所有操作
4. **无用户注册入口** — 唯一的用户创建方式是启动时 `seed_admin_user()` 自动创建的 admin

## 二、角色与权限设计

### 2.1 角色定义

| 角色 | 标识 | 说明 |
|------|------|------|
| 管理员 | `admin` | 拥有全部权限，可管理用户、系统配置 |
| 普通用户 | `user` | 可使用智能体对话、查看运行历史，不可管理后台资源 |

### 2.2 权限矩阵

| 功能模块 | admin | user |
|----------|-------|------|
| 智能体管理（增删改查、发布、复制、导入导出） | ✅ | ❌ |
| 知识库管理（增删改查、上传文档） | ✅ | ❌ |
| 工具管理（增删改查、测试） | ✅ | ❌ |
| 模型配置（增删改查） | ✅ | ❌ |
| 运行历史（查看、重试） | ✅ | ✅ 仅自己的 |
| 用户管理（增删改查、角色分配、禁用） | ✅ | ❌ |
| 前台对话（Portal） | ✅ | ✅ |
| 反馈提交 | ✅ | ✅ |

### 2.3 数据隔离

- `admin` 可查看所有运行历史
- `user` 只能查看自己创建的运行历史
- 后续可扩展：智能体、知识库等资源的创建者隔离

## 三、后端改动

### 3.1 角色枚举

在 `backend/app/core/config.py` 或新建 `backend/app/core/permissions.py` 中定义：

```python
class RoleEnum:
    ADMIN = "admin"
    USER = "user"

ROLE_PERMISSIONS = {
    RoleEnum.ADMIN: {"all"},
    RoleEnum.USER: {
        "conversation:read", "conversation:write",
        "run:read_own", "run:retry_own",
        "feedback:read", "feedback:write",
        "portal:access",
    },
}
```

### 3.2 权限守卫依赖

在 `backend/app/core/deps.py` 中新增：

```python
from functools import wraps

def require_role(*roles: str):
    """角色校验依赖，仅允许指定角色访问。"""
    async def _check(current: UserInfo = Depends(get_current_user)):
        if current.role not in roles:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail="权限不足",
            )
        return current
    return _check
```

使用方式：

```python
# 仅管理员可访问
@router.get("/users", dependencies=[Depends(require_role("admin"))])

# 管理员和普通用户均可访问
@router.get("/runs", dependencies=[Depends(require_role("admin", "user"))])
```

### 3.3 用户管理 API

新增 `backend/app/api/routes/users.py`：

| 接口 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `/api/users` | GET | admin | 用户列表（分页） |
| `/api/users` | POST | admin | 创建用户 |
| `/api/users/{id}` | PUT | admin | 编辑用户（角色、启用状态） |
| `/api/users/{id}` | DELETE | admin | 删除用户 |
| `/api/users/{id}/reset-password` | POST | admin | 重置用户密码 |

新增 Schema `backend/app/schemas/user.py`：

```python
class UserCreate(BaseModel):
    username: str = Field(min_length=1, max_length=80)
    password: str = Field(min_length=6)
    role: str = Field(default="user")

class UserUpdate(BaseModel):
    role: str | None = None
    is_active: bool | None = None

class UserOut(BaseModel):
    id: str
    username: str
    role: str
    is_active: bool
    created_at: datetime | None = None
```

### 3.4 现有路由添加权限守卫

| 路由文件 | 改动 |
|----------|------|
| `agents.py` | 添加 `require_role("admin")` |
| `knowledge.py` | 添加 `require_role("admin")` |
| `tools.py` | 添加 `require_role("admin")` |
| `settings.py` | 添加 `require_role("admin")` |
| `runs.py` | 保留 `get_current_user`，user 只能查看自己的记录 |
| `conversations.py` | 保持 `get_optional_user` 不变 |
| `feedback.py` | 保持现状 |

### 3.5 User 模型扩展

```python
class User(Base, TimestampMixin):
    __tablename__ = "users"

    id: Mapped[uuid.UUID] = mapped_column(primary_key=True, default=uuid.uuid4)
    username: Mapped[str] = mapped_column(String(80), unique=True, nullable=False)
    password_hash: Mapped[str] = mapped_column(String(255), nullable=False)
    role: Mapped[str] = mapped_column(String(30), default="user", nullable=False)  # 默认改为 user
    is_active: Mapped[bool] = mapped_column(Boolean, default=True, nullable=False)
```

> 注意：`role` 默认值从 `"admin"` 改为 `"user"`，`seed_admin_user()` 中显式设置 `role="admin"` 不受影响。

## 四、前端改动

### 4.1 Auth Store 扩展

`frontend/src/stores/auth.ts` 新增：

```typescript
getters: {
  isAdmin: (state) => state.user?.role === "admin",
}
```

### 4.2 菜单权限控制

在侧边栏/导航组件中，根据 `isAdmin` 控制菜单项显示：

- 管理员：显示全部菜单（智能体、知识库、工具、模型配置、用户管理）
- 普通用户：仅显示运行历史、前台入口

### 4.3 用户管理页面

新增 `frontend/src/pages/UsersPage.vue`：

- 用户列表（表格，含用户名、角色、状态、创建时间）
- 创建用户弹窗
- 编辑用户弹窗（修改角色、启用/禁用）
- 重置密码弹窗
- 删除确认

### 4.4 路由守卫增强

```typescript
router.beforeEach((to) => {
  const auth = useAuthStore();
  if (to.meta.requireAdmin && !auth.isAdmin) {
    return "/"; // 无权限跳转首页
  }
});
```

管理类路由添加 `meta: { requireAdmin: true }`。

## 五、数据库迁移

### 5.1 迁移脚本要点

1. 将现有用户的 `role` 保持为 `"admin"`（已有用户都是管理员创建的）
2. 新用户默认 `role` 改为 `"user"`
3. 新增 `users` 表的 CRUD 操作支持

### 5.2 兼容性

- 现有 admin 用户不受影响
- 现有 JWT Token 中已包含 `role`，无需重新登录即可生效（Token 中的 role 在签发时确定，过期前有效）
- 如需立即生效，可让用户重新登录

## 七、审计日志设计

### 7.1 audit_logs 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `user_id` | UUID | 操作者 ID（外键，关联 users） |
| `action` | VARCHAR(50) | 操作类型（create / update / delete / login / logout） |
| `resource_type` | VARCHAR(50) | 资源类型（agent / knowledge_base / tool / user / model_config） |
| `resource_id` | VARCHAR | 资源 ID |
| `detail` | JSON | 操作详情 |
| `ip_address` | VARCHAR(45) | 请求 IP |
| `created_at` | TIMESTAMP | 操作时间 |

### 7.2 记录范围

- 用户登录/退出
- 智能体创建/编辑/删除/发布/下线
- 知识库创建/删除/文档上传
- 工具创建/编辑/删除
- 模型配置创建/编辑/删除
- 用户管理操作（创建/编辑/禁用/角色变更）

## 八、密码策略

| 策略 | 规则 |
|------|------|
| 最小长度 | 8 位 |
| 复杂度 | 至少包含大写字母、小写字母和数字 |
| 哈希算法 | bcrypt，cost factor ≥ 12 |
| 登录失败锁定 | 5 次失败后锁定 15 分钟 |
| 首次登录 | 强制修改默认密码 |
| Token 有效期 | 建议缩短至 2-4 小时，引入 Refresh Token |

## 九、SSO / LDAP 集成预留

企业场景通常需要对接内部身份系统。建议在认证层预留扩展点：

1. **认证接口抽象**：将 `verify_password()` 抽象为 `AuthenticationProvider` 接口，支持多种认证方式。
2. **LDAP 配置**：新增 `ldap_server`、`ldap_bind_dn`、`ldap_search_base` 等配置项。
3. **OAuth 2.0 / OIDC**：预留 OAuth 2.0 授权码流程，支持企业微信、飞书、钉钉等第三方登录。
4. **用户同步**：定时从 LDAP/OIDC 同步用户信息，保持本地用户表与身份源一致。

## 十、权限变更审批

角色变更应有审批记录，防止权限滥用：

1. 角色变更请求记录到 `audit_logs` 表。
2. admin 角色变更需要另一个 admin 确认（双人确认机制）。
3. 权限提升（user → admin）需要审批流程。
4. 所有权限变更保留历史记录，不可删除。
